## 1. 核心套利逻辑

### 1.1 什么是 Polymarket 预测市场？

Polymarket 是一个去中心化预测市场平台。每个市场是一个二元期权：
- **YES 代币**（UP）：如果预测正确，到期值 $1
- **NO 代币**（DOWN）：如果预测错误，到期值 $1

价格区间 $0.01–$0.99，代表市场对该结果发生概率的集体判断。

### 1.2 本 Bot 的套利基础

本 bot 交易的是 **BTC 15 分钟涨跌预测市场**。每个市场持续 15 分钟，预测"在这 15 分钟内 BTC 是否会上涨"。

**套利的三个核心逻辑支柱：**

| # | 逻辑 | 说明 |
|---|------|------|
| 1 | **趋势跟踪**（主要） | Polymarket 的价格本身就是最好的信号。在 15 分钟窗口的第 13 分钟，价格 > $0.60 意味着市场高度确信 BTC 上涨，< $0.40 意味着下跌。此时买入相当于"跟随聪明钱" |
| 2 | **跨市场分歧**（辅助） | 当 Polymarket 价格与 Coinbase/Binance 现货价格出现背离时，存在定价错误。例如 BTC 现货在涨，但 Polymarket 价格仍低 → 买入 YES |
| 3 | **市场微观结构**（辅助） | Polymarket 订单簿失衡、报价变动速度、期权 PCR 等次级信号揭示短期方向 |

### 1.3 为什么这构成套利？

套利的本质是 **同一资产在不同市场间的价差**。这里的"资产"是"BTC 在 15 分钟内上涨的概率"：

- **Polymarket 价格** = 该概率的市场共识（$0.60 = 60% 概率）
- **外部信号** = 基于现货价格、订单簿、期权数据的独立概率估计

当两者出现可量化的分歧时 → 存在套利机会 → bot 在 Polymarket 上买入被低估的一方。

## 5. Phase 3 — Nautilus 交易框架

### 5.1 框架角色

NautilusTrader 是专业级交易框架，在本系统中负责：

| 组件 | 职责 |
|------|------|
| **Data Engine** | 处理 Polymarket 的实时 Quote Tick 流 |
| **Instrument Provider** | 加载 BTC 15-min 市场合约（YES/NO 代币对），按 slug 过滤 |
| **Event Dispatcher** | 事件驱动架构，处理订单事件、填充事件 |
| **Risk Engine** | Nautilus 内置风控（模拟模式下 bypass） |

### 5.2 市场加载机制

```
启动时 → 加载所有活跃 BTC 15-min 市场
        ↓
按 start_time 排序 → 找到"当前"市场（已开始且未结束）
        ↓
订阅该市场的 Quote Ticks → 实时接收买卖报价
        ↓
每 15 分钟 → 切换到下一个市场
```

关键算法 — 从 slug 提取时间戳：

```
slug = "btc-updown-15m-1718204400"
                         ↑
           Unix timestamp = 市场开始时间
```

## 6. Phase 4 — 信号处理层（套利信号生成）

这是整个系统的核心大脑。每个处理器都是一个独立的"套利视角"。

### 6.1 Spike Detection Processor（尖峰检测）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/spike_detector.py` |
| **算法** | 计算价格相对于 SMA-20 的偏离度 |
| **阈值** | `spike_threshold=0.05`（偏离超过 5% 视为尖峰） |
| **逻辑** | 价格突然大幅偏离近期均值 → 存在均值回归套利机会 |
| **套利类型** | **统计套利** — 价格偏离均值越远，回归概率越高 |
| **输出** | 大幅正偏离 → BEARISH（做空），大幅负偏离 → BULLISH（做多） |

```
price_deviation = (current_price - sma_20) / sma_20

if deviation > +0.05 → BEARISH (price too high, will revert down)
if deviation < -0.05 → BULLISH (price too low, will revert up)
```

### 6.2 Sentiment Processor（情绪分析）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/sentiment_processor.py` |
| **算法** | Fear & Greed 指数 + 消息面分析 |
| **阈值** | `extreme_fear_threshold=25`, `extreme_greed_threshold=75` |
| **逻辑** | 市场极度恐惧时 → 买入机会（反向指标） |
| **套利类型** | **反向套利** — 利用散户情绪的过度反应 |
| **权重** | 5%（信号较弱，仅作参考） |

### 6.3 Price Divergence Processor（价格背离）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/divergence_processor.py` |
| **算法** | 比较 Polymarket 价格 vs Coinbase 现货价格 |
| **阈值** | `divergence_threshold=0.05` |
| **逻辑** | Polymarket 价格与现货价格走势背离 → 存在定价错误 |
| **套利类型** | **跨市场套利** — 同一资产在两个市场定价不一致 |

```
BTC现货在涨 (Coinbase) 但 Polymarket价格低 → BULLISH (Polymarket定价偏低)
BTC现货在跌 (Coinbase) 但 Polymarket价格高 → BEARISH (Polymarket定价偏高)
```

### 6.4 OrderBook Imbalance Processor（订单簿失衡）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/orderbook_processor.py` |
| **算法** | 分析 Polymarket CLOB 订单簿的买卖力量对比 |
| **阈值** | `imbalance_threshold=0.30`, `min_book_volume=50.0` |
| **逻辑** | 买单深度显著大于卖单 → 短期上涨压力 |
| **套利类型** | **市场微观结构套利** — 利用订单簿揭示的短期供需失衡 |
| **权重** | 30%（权重最高，因为它是 Polymarket 自身的数据，最直接） |

```
imbalance = (bid_volume - ask_volume) / (bid_volume + ask_volume)

if imbalance > +0.30 → BULLISH (买单压倒性)
if imbalance < -0.30 → BEARISH (卖单压倒性)
```

### 6.5 Tick Velocity Processor（报价速度）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/tick_velocity_processor.py` |
| **算法** | 计算 Polymarket 价格在 30s/60s 窗口内的变动速度 |
| **阈值** | `velocity_threshold_60s=0.015`, `velocity_threshold_30s=0.010` |
| **逻辑** | 价格快速朝一个方向移动 → 动量将继续 |
| **套利类型** | **动量套利** — 趋势延续性 |
| **权重** | 25% |

```
velocity_60s = (price_now - price_60s_ago) / price_60s_ago

if velocity > +0.015 → BULLISH (快速上涨中)
if velocity < -0.015 → BEARISH (快速下跌中)
```

### 6.6 Deribit Put/Call Ratio Processor（期权 PCR）

| 项目 | 说明 |
|------|------|
| **文件** | `core/strategy_brain/signal_processors/deribit_pcr_processor.py` |
| **算法** | Deribit 交易所 BTC 期权 Put/Call 成交量和持仓量比率 |
| **阈值** | `bullish_pcr_threshold=1.20`, `bearish_pcr_threshold=0.70` |
| **逻辑** | 专业交易者的期权仓位揭示方向性观点 |
| **套利类型** | **衍生品套利** — 期权市场与预测市场价格分歧 |
| **权重** | 10% |

```
PCR = put_volume / call_volume

if PCR > 1.20 → BEARISH (大量 put = 专业交易者看跌)
if PCR < 0.70 → BULLISH (大量 call = 专业交易者看涨)
```

### 6.7 信号汇总

| 处理器 | 数据源 | 套利类型 | 权重 | 时效性 |
|---------|--------|----------|------|--------|
| Spike Detection | 自身价格历史 | 统计套利 | 12% | 中期 |
| Sentiment | Fear & Greed | 反向套利 | 5% | 慢 |
| Price Divergence | Coinbase 现货 | 跨市场套利 | 18% | 实时 |
| OrderBook Imbalance | Polymarket CLOB | 微观结构套利 | **30%** | 实时 |
| Tick Velocity | 自身报价流 | 动量套利 | **25%** | 实时 |
| Deribit PCR | Deribit 期权 | 衍生品套利 | 10% | 中速 |

## 7. Phase 4 — 融合引擎

**文件**: `core/strategy_brain/fusion_engine/signal_fusion.py`

### 7.1 加权投票机制

```
融合信号 = Σ(信号方向 × 权重 × 置信度) / Σ(权重)

最终方向 = BULLISH (score > 0) 或 BEARISH (score < 0)
最终分数 = |融合值| 归一化到 0-100
最终置信度 = 根据信号一致性和强度计算
```

### 7.2 参数

| 参数 | 值 | 说明 |
|------|----|------|
| `min_signals` | 1 | 最少需要 1 个信号触发 |
| `min_score` | 40.0 | 最低融合分数阈值 |

### 7.3 权重分布

| 处理器 | 权重 | 理由 |
|--------|------|------|
| OrderBookImbalance | 0.30 | 最直接的 Polymarket 供需数据 |
| TickVelocity | 0.25 | 短周期动量，在 15 分钟窗口有效 |
| PriceDivergence | 0.18 | 跨市场验证，但受限于 REST 延迟 |
| SpikeDetection | 0.12 | 均值回归，适合反转场景 |
| DeribitPCR | 0.10 | 机构观点，延迟较大 |
| SentimentAnalysis | 0.05 | 日频数据，仅做参考 |

## 8. Phase 5 — 执行与风控

### 8.1 趋势过滤器（核心决策层）

在 15 分钟市场的第 **13 分钟**（窗口 780s–840s），**趋势过滤器覆盖所有信号处理器**：

```
价格 > $0.60 → 市场说"涨" → 买入 YES (UP)
价格 < $0.40 → 市场说"跌" → 买入 NO (DOWN)
价格 $0.40–$0.60 → 盘整 → 不交易

为什么要等到第13分钟？
  第0-5分钟：噪音大，价格不确定
  第5-10分钟：趋势开始形成
  第13分钟：结果基本确定，价格充分反映了15分钟内的BTC走势
```

> **这是最重要的套利洞见**：在这个时间窗口，Polymarket 价格已经是一个高度准确的趋势信号。我们不是在预测未来——而是在**读取已经发生的趋势，并利用 Polymarket 的二元期权机制获利**。

### 8.2 风险引擎

**文件**: `execution/risk_engine.py`

| 限制 | 值 | 说明 |
|------|-----|------|
| 每笔最大 | **$1.00** | 严格的单笔风险上限 |
| 总敞口 | $10.00 | 同时最多 $10 风险 |
| 最大持仓 | 5 笔 | 同时最多 5 个活跃头寸 |
| 止损 | ~30% | 内置在下单逻辑中 |
| 止盈 | ~20% | 内置在下单逻辑中 |
| 最大回撤 | 15% | 触发后停止交易 |
| 日亏损上限 | $5.00 | 每日最大允许亏损 |

### 8.3 流动性检查

下单前检查 Polymarket 订单簿深度：
```
if 方向 == BUY 且 ask <= $0.02 → 流动性不足，跳过
if 方向 == SELL 且 bid <= $0.02 → 流动性不足，跳过
```

### 8.4 执行流程

```
信号 → 趋势过滤器 → 风控检查 → 流动性检查
        ↓              ↓            ↓
    通过/跳过     通过/拒绝     通过/跳过
        ↓
    下单 (模拟 or 真实)
        ↓
    Polymarket CLOB
```

**模拟模式**：`bot.py` 的 `_record_paper_trade()` 记录到 SQLite + JSON + 日志

**真实模式**：通过 Nautilus → PolymarketAdapter → CLOB API 下单

## 12. 风险说明

### 12.1 策略风险

| 风险 | 说明 | 缓释措施 |
|------|------|----------|
| **趋势反转** | 第 13 分钟买入后最后 2 分钟反转 | 严格 $1 封顶，单次损失可控 |
| **流动性枯竭** | Polymarket 订单簿深度不足 | 流动性检查，bid/ask < $0.02 跳过 |
| **信号失效** | 所有信号处理器同时给出错误方向 | 趋势过滤器作为最终裁决 |
| **Black Swan** | 极端行情导致市场异常 | 15% 最大回撤硬止损 |
| **API 故障** | Polymarket CLOB 不可用 | 跳过本轮交易，下次重试 |

### 12.2 资金管理

```
初始资本：$1,000
```

### 12.3 关键警告

> **这不是无风险套利。** 虽然我们称之为"套利"，但本系统本质上是**统计套利 + 趋势跟踪**的组合策略。不存在锁定的无风险利润。Polymarket 的二元期权机制意味着每笔交易要么赢要么输——没有中间状态。

> **过去表现不代表未来收益。** 模拟测试的高胜率可能无法在真实市场中复现。
